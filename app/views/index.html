Performance Platform changelog

Prototypes created during the development of the Performance Platform since the re-discovery in early 2016.

If you need the username and password to these, you'll need to get in touch with the team.

PP 5  Data filtersALPHA

January 2016

Main changes
Change Why we made it
View by department, agency or service. Ability to filter and sort by any metric. Different users have very different interests depending on what they are working on. We think a new overview page will give users more flexibilty, allowing them to create customised filters of the data.
Clearer relationship between numbers. Users were often unclear about which numbers were part of which, for example how calls received related to transactions received by phone. A new page layout tries to clarify this.
Data guidance in context. As we're introducing new concepts and terms, we need to explain what they mean. Instead of having a separate 'guidance' section, we added information next to where the data is to save users clicking back and forth and reading long pages of text.
Used 'view all' as primary call to action, moved search to separate page. Previous testing showed that users benefitted from seeing what data is available on the website, and are only likely to search in specific situations. We think that the need to search for something specific could be adequately met on a separate page.

PP 4  Value proposition & user effortALPHA

November 2016

Main changes
Change Why we made it
Designed a new 'product page' with a clearer value proposition We found that users had a varying understanding of what the site offers them. We also moved to be more consistent with the product pages being used by other GDS products.
Reformed 'user effort' into 'calls' and 'casework'. We found that putting calls and casework data together confused users. We think that making each into a distinct data set (and removing the 'user effort' name) will clarify this.

PP 3  Initial MVPALPHA

October 2016

Main changes
Change Why we made it
Add a search function with autocomplete to the home page. We found in research that most participants wanted to be able to search for something specific. We think an autocomplete will help suggest what it's possible to search for.
Added user effort data, broken down by 'reasons users get in touch' and 'reasons users have to repeat or amend something'. To test whether this kind of information is useful for our users to assess service quality.
Removed transactional service ('task') pages. To test our hypothesis that our users only want to see the data at a higher level (i.e. department, agency and so on).

PP 2  Grouping and aggregationDISCO ALPHA

August 2016

Main changes
Change Why we made it
Design a clickable interface for browsing the data. We found that the spreadsheet in the last iteration was overwhelming or too advanced for most users (although some need the ability to download in CSV format too).
Show aggregated data by department, agency, topic and 'sub-topic'. We think that grouping data in different ways will allow our users to get a higher-level view of what's going on across government.
Add a filter/comparison tool. We think that users may need to create their own views of the data by combining different departments or areas of interest.
