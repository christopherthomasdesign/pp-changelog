Publish data prototypes

Service blueprint v1  Improving the on-boarding processBETA

January 2018

Main changes
  • We built the web form part of the prototype tested during Alpha and started using it to get data from services. We then mapped out the entire end-to-end process for data publishers and sought to make improvements to it.
  • The biggest change was the introduction of a 'kick-off meeting' in which the Service Performance team met with new data publishers to go through the metrics. This was intended to increase motivation to publish data, set expectations, and make sure that users' interpretation of the metrics is correct.

Iteration 4  Web forms sent direct to service teamsALPHA

August 2017

Main changes
Change Why we made it
Re-focused to get data directly from a contact on a service team We think that by going direct to services, we can communicate more easily, motivate users to give data, and remove a lot of redundant users from the data gathering process.
Use web forms to gather data, sent to publishers via automated emails We've found in prior research that web forms work better for inputting a small amount of data. They can be used by anyone. They cannot be emailed around and reformatted like spreadsheets are.

Iteration 3  Automating sending spreadsheets aroundALPHA

July 2017

Main changes
Change Why we made it
Narrow scope to focus on people who gather data for multiple services on behalf of their organisation. This is how most data is currently published to the Performance Platform. We think that be easing some of the pain points for these central contacts, we can get better data more regularly. We should also be able to make a smoother transition from one product to another.
Use email and upload links to automate the process of asking for and gathering data. To remove the pain point of forwarding and receiving loads of emails with spreadsheets of data. The central contact's role becomes more focused on monitoring and sense-checking.
Create an account to manage data from. To see there is a need to have somewhere to manage this, and if it's any better than email.

Iteration 2  Publish manually or automaticallyALPHA

June 2017

Main changes
Change Why we made it
Test a new option to explore automatic publishing of data via APIs. We've found from users that automatic publishing is the ideal scenario, but hard to get set up for certain services/organisations. We think that offering a more hands-on service to get users started will make this easier.
Group the previously tested ideas into one, with two routes for different responsibilities. To test if users knew whether they were responsible for one or more services. We also need to test the previous prototypes with more users who publish data for multiple services so we can be sure of our findings.

Iteration 1  Publish for services or departmentsALPHA

June 2017

Two options to test different levels of responsibility:

  • One person responsible for one service, with ability to delegate responsibility for individual metrics
  • One person responsible for one department, with ability to delegate responsibility for individual services
